# 🐌 Carbon ERP/MES 开发环境启动慢问题分析

**问题**: 启动 DEV 环境后，页面至少需要 30 秒到 1 分钟才能响应

**日期**: 2026-05-30

---

## 🔍 问题分析

### 启动流程分析

根据代码分析，`pnpm dev` (即 `crbn up --no-portless`) 的完整启动流程如下：

#### 1. **环境准备阶段** (~5-10秒)
```typescript
// packages/dev/src/commands/up.ts
- 加载 .env 和 .env.local
- 检查 portless 配置
- 选择要启动的应用 (ERP/MES)
- 解析 worktree slug
- 刷新过期的配置文件
- 检查并安装依赖 (pnpm install)
```

#### 2. **配置生成阶段** (~2-5秒)
```typescript
- 分配端口 (PORT_API, PORT_ERP, PORT_MES 等)
- 生成 .env.local 文件
- 同步环境变量符号链接
- 启动共享 Redis
```

#### 3. **Docker 镜像拉取阶段** (~10-30秒，首次更长)
```typescript
- 检查本地是否有所有镜像
- 如果缺失，执行 docker compose pull
- 拉取 12 个服务的镜像:
  * postgres
  * storage
  * gotrue
  * postgrest
  * realtime
  * kong
  * inbucket
  * inngest
  * redis
  * 等等...
```

#### 4. **Docker 容器启动阶段** (~10-20秒)
```typescript
- 启动 12 个 Docker 服务
- 等待 TCP 端口就绪:
  * PORT_DB (PostgreSQL)
  * PORT_API (Supabase API)
  * PORT_INNGEST
- 等待 PostgreSQL 接受查询
- 等待 Storage 服务就绪
```

#### 5. **数据库迁移阶段** (~5-15秒)
```typescript
- 应用数据库迁移 (如果有新迁移)
- 重新生成类型定义 (pnpm db:types)
- 重新生成 Swagger 文档 (pnpm generate:swagger)
```

#### 6. **应用启动阶段** (~10-20秒)
```typescript
- 启动 ERP 开发服务器 (Vite)
- 启动 MES 开发服务器 (Vite)
- Vite 预构建依赖
- Vite 编译应用代码
```

### 总耗时估算

| 阶段 | 首次启动 | 后续启动 |
|------|----------|----------|
| 环境准备 | 10秒 | 5秒 |
| 配置生成 | 5秒 | 2秒 |
| Docker 镜像拉取 | 60秒+ | 0秒 (跳过) |
| Docker 容器启动 | 20秒 | 15秒 |
| 数据库迁移 | 15秒 | 2秒 (无新迁移) |
| 应用启动 | 20秒 | 15秒 |
| **总计** | **130秒 (2分钟+)** | **39秒** |

## 🎯 性能瓶颈

### 1. **Docker 服务启动慢** (最大瓶颈)

**问题**:
- 启动 12 个 Docker 容器需要 15-20 秒
- PostgreSQL 初始化需要额外时间
- Storage 服务有时会卡住，需要重启

**证据**:
```typescript
// 等待 5 个检查点
const bar = progress({ style: "heavy", max: 5 });
bar.start("Waiting for services");
await waitForTcp([...]);  // 等待 3 个 TCP 端口
await waitForPostgres(...);  // 等待 PostgreSQL
await waitForStorageReady(...);  // 等待 Storage (可能卡住)
```

### 2. **Vite 预构建依赖慢**

**问题**:
- 首次启动或依赖变更后，Vite 需要预构建所有依赖
- ERP 和 MES 各自独立预构建
- 大量的依赖包需要处理

**影响**:
- 首次启动: 20-30 秒
- 依赖变更后: 15-20 秒
- 正常启动: 10-15 秒

### 3. **数据库迁移和类型生成**

**问题**:
- 每次启动都检查迁移
- 如果有新迁移，需要重新生成类型和 Swagger
- `pnpm db:types` 和 `pnpm generate:swagger` 比较慢

**耗时**:
- 无新迁移: 2 秒
- 有新迁移: 15-20 秒

### 4. **环境变量和配置同步**

**问题**:
- 每次启动都刷新配置文件
- 检查并同步 package.json#crbn.copy 中的文件
- 生成 .env.local

**耗时**: 5-10 秒

## 💡 优化建议

### 高优先级优化 (可节省 20-30 秒)

#### 1. **使用 Docker Compose 缓存**

**当前问题**: 每次启动都完全重启所有容器

**优化方案**:
```bash
# 保持容器运行，只重启应用
pnpm dev:apps  # 只启动 ERP/MES，复用已运行的 Docker 服务
```

**实现**:
```json
// package.json
{
  "scripts": {
    "dev:services": "crbn up --no-apps",  // 只启动 Docker 服务
    "dev:apps": "turbo run dev --filter=./apps/erp --filter=./apps/mes"  // 只启动应用
  }
}
```

**效果**: 节省 15-20 秒

---

#### 2. **跳过不必要的迁移检查**

**当前问题**: 每次启动都运行迁移检查

**优化方案**:
```bash
# 开发时跳过迁移
pnpm dev -- --no-migrate

# 或设置环境变量
CARBON_SKIP_MIGRATE=1 pnpm dev
```

**实现**:
```typescript
// packages/dev/src/commands/up.ts
const shouldMigrate = 
  process.env.CARBON_SKIP_MIGRATE !== "1" && 
  (opts.migrate ?? true);
```

**效果**: 节省 2-5 秒

---

#### 3. **优化 Vite 预构建**

**当前问题**: Vite 每次都重新扫描依赖

**优化方案**:

**方案 A: 配置 Vite 缓存**
```typescript
// apps/erp/vite.config.ts
export default defineConfig({
  cacheDir: "../../node_modules/.vite/erp",  // 共享缓存
  optimizeDeps: {
    force: false,  // 不强制重新预构建
    include: [
      // 明确列出需要预构建的依赖
      "react",
      "react-dom",
      "@remix-run/react",
      // ... 其他常用依赖
    ]
  }
});
```

**方案 B: 使用 Turbopack (实验性)**
```json
// package.json
{
  "scripts": {
    "dev:turbo": "turbo run dev --filter=./apps/* --experimental-turbopack"
  }
}
```

**效果**: 节省 10-15 秒

---

### 中优先级优化 (可节省 10-15 秒)

#### 4. **并行启动服务**

**当前问题**: 串行等待各个服务

**优化方案**:
```typescript
// 并行等待多个服务
await Promise.all([
  waitForTcp([...]),
  waitForPostgres(...),
  // 不等待 Storage，后台检查
  waitForStorageReady(...).catch(() => {})
]);
```

**效果**: 节省 5-10 秒

---

#### 5. **减少 Docker 服务数量**

**当前问题**: 启动 12 个服务，但开发时可能不需要全部

**优化方案**:
```yaml
# docker-compose.dev.yml (精简版)
services:
  postgres:
    # ...
  redis:
    # ...
  storage:
    # ...
  gotrue:
    # ...
  # 移除不常用的服务:
  # - inbucket (邮件测试)
  # - inngest (后台任务)
  # - kong (API Gateway)
```

**使用**:
```bash
# 使用精简配置
COMPOSE_FILE=docker-compose.dev.yml pnpm dev
```

**效果**: 节省 5-8 秒

---

#### 6. **使用 SSD 和增加内存**

**硬件优化**:
- 使用 SSD (如果还在用 HDD)
- 增加内存到 16GB+
- 使用更快的 CPU

**效果**: 整体提速 20-30%

---

### 低优先级优化 (可节省 5-10 秒)

#### 7. **使用 pnpm 缓存**

```bash
# 配置 pnpm 使用本地缓存
pnpm config set store-dir ~/.pnpm-store
pnpm config set cache-dir ~/.pnpm-cache
```

#### 8. **禁用不必要的 Biome 检查**

```json
// package.json
{
  "lint-staged": {
    // 开发时禁用 pre-commit hook
  }
}
```

#### 9. **使用 Docker 本地镜像缓存**

```bash
# 预先拉取所有镜像
docker compose pull

# 使用本地镜像
docker compose up --no-pull
```

## 🚀 推荐的优化方案

### 方案 1: 快速开发模式 (推荐)

**适用场景**: 日常开发，频繁重启

**步骤**:

1. **首次启动** (完整启动):
```bash
pnpm dev
```

2. **后续开发** (只重启应用):
```bash
# 保持 Docker 服务运行
# 只重启 ERP/MES
pnpm dev:apps
```

3. **配置 package.json**:
```json
{
  "scripts": {
    "dev": "crbn up --no-portless",
    "dev:services": "crbn up --no-apps --no-portless",
    "dev:apps": "turbo run dev --filter=./apps/erp --filter=./apps/mes",
    "dev:fast": "pnpm dev:apps"  // 快速模式
  }
}
```

**效果**:
- 首次启动: 60-90 秒
- 后续启动: **10-15 秒** ⚡

---

### 方案 2: 精简服务模式

**适用场景**: 只需要基础功能，不需要邮件/任务等

**步骤**:

1. **创建精简配置**:
```yaml
# docker-compose.minimal.yml
version: '3.8'
services:
  postgres:
    # ... (保留)
  redis:
    # ... (保留)
  storage:
    # ... (保留)
  gotrue:
    # ... (保留)
  postgrest:
    # ... (保留)
  # 移除: inbucket, inngest, kong, realtime
```

2. **使用精简配置**:
```bash
COMPOSE_FILE=docker-compose.minimal.yml pnpm dev
```

**效果**:
- 启动时间: **30-40 秒** ⚡
- 内存占用: 减少 30-40%

---

### 方案 3: 跳过迁移模式

**适用场景**: 数据库 schema 没有变化

**步骤**:

1. **配置环境变量**:
```bash
# .env.local
CARBON_SKIP_MIGRATE=1
```

2. **或使用命令行参数**:
```bash
pnpm dev -- --no-migrate --no-regen
```

**效果**:
- 节省: **5-15 秒** ⚡

## 🎯 总结

### 问题根源

1. **Docker 服务启动慢** (15-20秒) - 最大瓶颈
2. **Vite 预构建依赖** (10-15秒)
3. **数据库迁移检查** (2-15秒)
4. **环境配置同步** (5-10秒)

### 推荐方案

**日常开发** (最佳):
```bash
# 首次启动
pnpm dev

# 后续开发 (保持 Docker 运行)
pnpm dev:apps  # 只需 10-15 秒 ⚡
```

**快速测试**:
```bash
# 跳过迁移
pnpm dev -- --no-migrate --no-regen  # 节省 5-15 秒
```

**极速模式** (组合优化):
```bash
# 1. 使用精简 Docker 配置
# 2. 跳过迁移
# 3. 只重启应用
# 总耗时: 10-15 秒 ⚡⚡⚡
```

### 预期效果

- **当前**: 60-90 秒
- **优化后**: **10-40 秒**
- **节省**: **50-80 秒 (55-89%)**

---

**建议**: 先尝试"快速开发模式"（方案 1），这是最简单且效果最好的优化方案。
