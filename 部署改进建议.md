## 🤔 为什么 ERP 和 MES 需要两个端口？

### 架构设计原因

Carbon 采用 **单体数据库 + 多前端应用** 的架构：

```
┌─────────────────────────────────────────────────────────┐
│              Supabase PostgreSQL (统一数据库)            │
│                                                         │
│  • 公司、用户、权限                                      │
│  • 零件、订单、库存                                      │
│  • 生产任务、工序、质量                                  │
│  • 所有业务数据共享                                      │
└─────────────────────────────────────────────────────────┘
           ↑                           ↑
           │                           │
    ┌──────┴──────┐           ┌───────┴────────┐
    │  ERP 应用   │           │   MES 应用     │
    │  :4000      │           │   :4001        │
    │             │           │                │
    │ React Router│           │ React Router   │
    │ 独立构建    │           │ 独立构建       │
    └─────────────┘           └────────────────┘
```

### 为什么是两个独立应用？

#### 1. **用户群体不同**

| 特性 | ERP | MES |
|-----|-----|-----|
| **用户** | 办公室人员（销售、采购、财务） | 车间人员（操作员、质检员） |
| **设备** | 电脑、笔记本 | 平板、触摸屏、工业终端 |
| **环境** | 办公室 | 车间（可能有油污、灰尘） |
| **操作** | 鼠标键盘 | 触摸、扫码枪 |

#### 2. **UI 设计差异**

**ERP UI**:
- 📊 表格为主，数据密集
- 🖱️ 复杂表单和筛选
- 📈 丰富的报表和图表
- 🎯 功能全面

**MES UI**:
- 📱 大按钮、卡片式
- 👆 触摸友好
- ⏱️ 实时更新、状态指示
- 🎯 聚焦执行

#### 3. **性能优化**

```javascript
// apps/erp/package.json
{
  "name": "erp",
  "dependencies": {
    // ERP 特有的库
    "@tanstack/react-table": "...",  // 复杂表格
    "recharts": "...",                // 图表
    "xlsx": "..."                     // Excel 导出
  }
}

// apps/mes/package.json
{
  "name": "mes",
  "dependencies": {
    // MES 特有的库
    "react-qr-reader": "...",         // 扫码
    "react-webcam": "...",            // 摄像头
    // 更轻量的依赖
  }
}
```

**好处**:
- ✅ ERP 不需要加载扫码、摄像头等 MES 功能
- ✅ MES 不需要加载复杂的报表、Excel 导出等 ERP 功能
- ✅ 各自的 bundle 更小，加载更快

#### 4. **独立部署和扩展**

```
生产环境示例：

办公室（10 人）:
  ERP: 1 台服务器 (2核4GB)
  
车间（50 人，3 班倒）:
  MES: 3 台服务器 (4核8GB) + 负载均衡
  
数据库:
  PostgreSQL: 1 台服务器 (8核16GB)
```

**好处**:
- ✅ 可以根据实际使用情况独立扩展
- ✅ MES 高峰期不影响 ERP
- ✅ 可以分别部署到不同的网络区域

#### 5. **安全隔离**

```
网络架构：

办公网络 (192.168.1.0/24)
  ├── ERP: https://erp.company.com
  └── 数据库: 内网访问
  
车间网络 (192.168.2.0/24)
  ├── MES: https://mes.company.com
  └── 数据库: 内网访问
```

**好处**:
- ✅ 车间网络可以限制只访问 MES
- ✅ 办公网络可以限制只访问 ERP
- ✅ 减少攻击面

### 数据是如何共享的？

虽然是两个应用，但数据完全互通：

```typescript
// ERP 创建生产任务
const { data } = await supabase
  .from('job')
  .insert({
    companyId: 'company-123',
    partId: 'part-456',
    quantity: 100
  });

// MES 立即可以看到（实时同步）
supabase
  .channel('jobs')
  .on('postgres_changes', 
    { event: 'INSERT', schema: 'public', table: 'job' },
    (payload) => {
      // MES 界面自动更新
      console.log('新任务:', payload.new);
    }
  )
  .subscribe();
```

**关键点**:
- ✅ 共享同一个 PostgreSQL 数据库
- ✅ 使用 Supabase Realtime 实时同步
- ✅ 通过 `companyId` 隔离不同公司数据
- ✅ Row-Level Security (RLS) 确保数据安全

## 🔍 当前部署的不足

### 1. 缺少生产环境配置

**问题**:
- ❌ 只有开发环境配置 (`dev.sh`)
- ❌ 没有生产环境的 Docker Compose 配置
- ❌ 没有生产环境的环境变量模板

**影响**:
- 部署到服务器需要手动配置
- 容易出错
- 不易维护

### 2. 缺少反向代理配置

**问题**:
- ❌ 没有 Nginx 配置模板
- ❌ 没有 SSL 证书配置说明
- ❌ 没有负载均衡配置

**影响**:
- 需要手动配置 Nginx
- 容易配置错误
- 不支持 HTTPS

### 3. 缺少进程管理

**问题**:
- ❌ 没有 PM2 配置文件
- ❌ 没有 systemd 服务配置
- ❌ 应用崩溃后不会自动重启

**影响**:
- 应用不稳定
- 需要手动重启
- 无法监控应用状态

### 4. 缺少监控和日志

**问题**:
- ❌ 没有日志收集配置
- ❌ 没有监控告警配置
- ❌ 没有性能监控

**影响**:
- 问题难以排查
- 无法及时发现故障
- 无法优化性能

### 5. 缺少备份方案

**问题**:
- ❌ 没有数据库备份脚本
- ❌ 没有自动备份配置
- ❌ 没有备份恢复说明

**影响**:
- 数据丢失风险
- 无法快速恢复
- 业务中断

#### 4. Nginx 配置

```nginx
# deployment/nginx/sites/erp.conf
upstream erp_backend {
    least_conn;
    server 127.0.0.1:4000;
    keepalive 64;
}

server {
    listen 80;
    server_name erp.company.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name erp.company.com;

    ssl_certificate /etc/letsencrypt/live/erp.company.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/erp.company.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    access_log /var/log/nginx/erp-access.log;
    error_log /var/log/nginx/erp-error.log;

    client_max_body_size 100M;

    location / {
        proxy_pass http://erp_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
        proxy_read_timeout 300s;
        proxy_connect_timeout 75s;
    }

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
        proxy_pass http://erp_backend;
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

#### 5. 自动备份脚本

```bash
#!/bin/bash
# deployment/scripts/backup-db.sh

BACKUP_DIR="/var/backups/carbon"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/carbon_$DATE.sql"

# 创建备份目录
mkdir -p $BACKUP_DIR

# 备份数据库
docker exec supabase-db pg_dump -U postgres postgres > $BACKUP_FILE

# 压缩备份
gzip $BACKUP_FILE

# 删除 7 天前的备份
find $BACKUP_DIR -name "*.sql.gz" -mtime +7 -delete

echo "备份完成: $BACKUP_FILE.gz"
```

#### 6. 健康检查脚本

```bash
#!/bin/bash
# deployment/scripts/health-check.sh

# 检查 ERP
ERP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://erp.company.com)
if [ $ERP_STATUS -eq 200 ]; then
    echo "✅ ERP 正常"
else
    echo "❌ ERP 异常 (HTTP $ERP_STATUS)"
    exit 1
fi

# 检查 MES
MES_STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://mes.company.com)
if [ $MES_STATUS -eq 200 ]; then
    echo "✅ MES 正常"
else
    echo "❌ MES 异常 (HTTP $MES_STATUS)"
    exit 1
fi

# 检查数据库
DB_STATUS=$(docker exec supabase-db pg_isready -U postgres)
if [ $? -eq 0 ]; then
    echo "✅ 数据库正常"
else
    echo "❌ 数据库异常"
    exit 1
fi

echo "✅ 所有服务正常"
```

## 📝 具体实施步骤

### 第一阶段：准备配置文件（1-2 天）

1. **创建部署目录结构**
   ```bash
   mkdir -p deployment/{nginx/sites,pm2,systemd,scripts,monitoring}
   ```

2. **编写配置文件**
   - Docker Compose 生产配置
   - Nginx 配置
   - PM2 配置
   - 备份脚本
   - 部署脚本

3. **测试配置文件**
   - 在本地测试 Docker Compose
   - 验证 Nginx 配置语法
   - 测试备份脚本

### 第二阶段：服务器准备（1 天）

1. **购买阿里云 ECS**
   - 配置：4核8GB, 100GB SSD
   - 地域：选择离你最近的区域
   - 操作系统：Ubuntu 22.04 LTS

2. **配置域名**
   - 购买域名（如 company.com）
   - 添加 DNS 解析：
     - erp.company.com → 服务器 IP
     - mes.company.com → 服务器 IP
     - api.company.com → 服务器 IP

3. **安全配置**
   - 配置安全组（开放 80, 443, 22）
   - 配置 SSH 密钥登录
   - 禁用密码登录

### 第三阶段：部署应用（2-3 天）

1. **安装基础软件**
   ```bash
   # Docker
   curl -fsSL https://get.docker.com | sh
   
   # Node.js & pnpm
   curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
   sudo apt install -y nodejs
   npm install -g pnpm pm2
   
   # Nginx
   sudo apt install -y nginx
   ```

2. **部署 Supabase**
   ```bash
   git clone https://github.com/supabase/supabase
   cd supabase/docker
   cp .env.example .env
   # 编辑 .env 配置
   docker compose up -d
   ```

3. **部署 Carbon 应用**
   ```bash
   git clone your-carbon-repo.git
   cd carbon
   cp deployment/.env.prod.example .env
   # 编辑 .env 配置
   pnpm install
   pnpm build
   pnpm db:migrate
   pm2 start deployment/pm2/ecosystem.config.js
   ```

4. **配置 Nginx**
   ```bash
   sudo cp deployment/nginx/sites/*.conf /etc/nginx/sites-available/
   sudo ln -s /etc/nginx/sites-available/*.conf /etc/nginx/sites-enabled/
   sudo nginx -t
   sudo systemctl reload nginx
   ```

5. **配置 SSL**
   ```bash
   sudo apt install -y certbot python3-certbot-nginx
   sudo certbot --nginx -d erp.company.com -d mes.company.com -d api.company.com
   ```

### 第四阶段：配置监控和备份（1 天）

1. **配置自动备份**
   ```bash
   # 添加 cron 任务
   crontab -e
   # 每天凌晨 2 点备份
   0 2 * * * /path/to/deployment/scripts/backup-db.sh
   ```

2. **配置 PM2 监控**
   ```bash
   pm2 install pm2-logrotate
   pm2 set pm2-logrotate:max_size 100M
   pm2 set pm2-logrotate:retain 7
   ```

3. **配置阿里云监控**
   - 在阿里云控制台配置监控告警
   - 设置 CPU、内存、磁盘告警阈值

### 第五阶段：测试和优化（1-2 天）

1. **功能测试**
   - 测试 ERP 所有功能
   - 测试 MES 所有功能
   - 测试数据同步

2. **性能测试**
   - 使用 Apache Bench 或 JMeter 压测
   - 优化数据库查询
   - 调整 Nginx 和 PM2 配置

3. **安全测试**
   - 检查 SSL 配置
   - 检查防火墙规则
   - 检查数据库权限

## ✅ 总结

### 为什么需要两个端口？

1. **用户群体不同** - 办公室 vs 车间
